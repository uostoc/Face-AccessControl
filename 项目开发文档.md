# 人脸识别门禁与晚归管理系统开发文档

## 1. 项目概述

本项目用于书院场景下的人员识别、陌生人员发现和晚归事件辅助管理。系统通过摄像头采集视频流，对画面中的人脸进行检测、对齐、特征提取和人员库比对，最终生成通行日志、陌生人员事件和晚归疑似记录。

项目定位为“辅助识别与管理系统”，不建议在初期直接作为强制处罚或自动处置依据。所有低置信度、陌生人、晚归疑似事件应支持管理员复核。

## 2. 建设目标

### 2.1 核心目标

- 识别书院合法人员。
- 发现非书院人员或未知人员。
- 记录人员在指定区域的出现时间、地点和抓拍证据。
- 在门禁时间后生成晚归疑似记录。
- 提供管理员复核、查询、导出和人员库维护能力。

### 2.2 非目标

- 初期不做自动处分。
- 初期不强依赖远程推送。
- 初期不直接接管物理门禁开关。
- 初期不承诺 100% 准确识别所有侧脸、遮挡、口罩、弱光场景。

## 3. 业务范围

### 3.1 非书院人员识别

系统检测到人脸后，与人员库进行比对。

- 高置信匹配：记录正常通行。
- 低置信匹配：进入观察或待复核。
- 多帧确认未知：生成陌生人员事件。
- 高风险场景下可触发本地 LED 或蜂鸣器提示。

### 3.2 晚归人员识别

系统在门禁时间后识别到人员时，生成晚归相关记录。

推荐分阶段实现：

- V1：仅生成“晚归疑似记录”，由管理员复核。
- V2：接入门禁出入记录，判断是否属于“外出后超时返回”。
- V3：通过双摄像头或方向识别判断进出方向。

### 3.3 人员库管理

支持常驻学生、老师、管理员、访客、失效人员的统一维护。

人员状态：

- `active`：有效人员。
- `visitor`：临时访客。
- `expired`：已过期人员。
- `graduated`：已毕业人员。
- `disabled`：停用人员。

## 4. 总体架构

```text
摄像头 / 机器人
    ↓
视频采集服务
    ↓
人脸检测与质量评估
    ↓
人脸对齐与特征提取
    ↓
人员库比对
    ↓
事件判定引擎
    ↓
SQLite 数据库 / 文件存储
    ↓
管理端 / 日志 / 告警 / 报表
```

## 5. 技术选型

| 模块 | 推荐方案 | 说明 |
|---|---|---|
| 开发语言 | Python | 适合 OpenCV、InsightFace、模型推理 |
| 人脸识别框架 | InsightFace | 集成检测、关键点、特征提取 |
| 模型包 | `buffalo_l` | 初期综合效果较好 |
| 推理引擎 | onnxruntime | InsightFace 常用依赖 |
| 视频处理 | opencv-python | 摄像头采集、图像处理 |
| 数值计算 | numpy | 向量计算、余弦相似度 |
| 数据库 | SQLite | 初期比 JSON 更适合审计、查询和维护 |
| 后期向量检索 | FAISS | 人数规模增大后引入 |
| 管理端 | FastAPI + Web 前端 | 可分阶段建设 |

## 6. 模块划分

### 6.1 视频采集模块

职责：

- 打开 USB 摄像头、网络摄像头或机器人视频流。
- 按配置读取视频帧。
- 支持摄像头离线重连。
- 记录摄像头状态。

关键配置：

```yaml
camera:
  source: 0
  fps: 30
  width: 1280
  height: 720
  reconnect_interval_seconds: 5
```

### 6.2 人脸检测模块

职责：

- 检测画面中的所有人脸。
- 输出人脸框、关键点、检测置信度。
- 过滤过小、模糊、角度过大的无效人脸。

建议策略：

- 检测频率控制在 5 到 10 FPS。
- 中间帧使用跟踪结果复用。
- 同一帧只处理面积最大的前 N 张人脸。

### 6.3 人脸质量评估模块

职责：

- 判断人脸是否适合用于识别。

质量指标：

- 人脸面积。
- 清晰度。
- 亮度。
- 姿态角。
- 遮挡情况。
- 是否单人脸。

低质量人脸不直接判定为陌生人，只记录为“质量不足”或等待下一帧。

### 6.4 特征提取模块

职责：

- 对合格人脸进行对齐。
- 提取 512 维特征向量。
- 对向量进行归一化。

输出：

```text
embedding: float[512]
quality_score: float
face_image_path: string
```

### 6.5 人员库比对模块

职责：

- 从数据库加载有效人员特征。
- 使用余弦相似度计算最相似人员。
- 返回匹配结果、相似度和候选人员。

初始阈值：

| 相似度 | 判定 |
|---|---|
| `>= 0.60` | 确认匹配 |
| `0.40 ~ 0.60` | 疑似匹配 |
| `< 0.40` | 未匹配 |

阈值必须支持配置，并在现场试运行后校准。

### 6.6 多帧投票模块

职责：

- 降低单帧误识别。
- 对同一个跟踪目标累计多次识别结果。

推荐规则：

```text
连续 5 次有效识别中，至少 3 次匹配同一人员，确认身份。
连续 5 次有效识别中，至少 3 次为未知，生成陌生人员事件。
```

### 6.7 事件判定模块

职责：

- 将识别结果转换为业务事件。

事件类型：

- 正常通行。
- 疑似匹配。
- 陌生人员。
- 晚归疑似。
- 质量不足。
- 摄像头异常。

### 6.8 告警模块

职责：

- 对高风险事件进行本地提示或管理端提示。

初期建议：

- 正常通行：只记录日志。
- 疑似匹配：进入复核列表。
- 陌生人员：记录事件，可 LED 提示。
- 高风险陌生人员：LED + 蜂鸣器。

不建议初期对所有陌生人直接蜂鸣器报警。

### 6.9 管理端模块

最低功能：

- 人员列表。
- 人员新增、编辑、停用。
- 注册照上传或采集。
- 人脸特征生成。
- 通行日志查询。
- 陌生人员事件复核。
- 晚归疑似事件复核。
- 阈值配置。
- 摄像头状态查看。
- 数据导出。

## 8. 核心流程

### 8.1 人员注册流程

```text
录入人员基础信息
    ↓
采集正脸、左侧、右侧照片
    ↓
人脸质量检测
    ↓
质量合格则提取特征
    ↓
写入人员表和特征表
    ↓
注册完成
```

### 8.2 实时识别流程

```text
读取视频帧
    ↓
人脸检测
    ↓
质量评估
    ↓
合格人脸提取特征
    ↓
人员库相似度比对
    ↓
多帧投票
    ↓
生成识别日志
    ↓
按规则生成业务事件
```

### 8.3 陌生人员事件流程

```text
检测到未知人脸
    ↓
确认人脸质量合格
    ↓
多帧投票仍为未知
    ↓
检查是否短时间重复事件
    ↓
生成陌生人员事件
    ↓
本地提示或管理员复核
```

### 8.4 晚归事件流程

```text
识别到有效人员
    ↓
获取当前时间
    ↓
判断是否超过门禁时间和宽限期
    ↓
检查短时间内是否已记录
    ↓
生成晚归疑似事件
    ↓
管理员复核
```

## 12. 告警策略

### 12.1 分级规则

| 等级 | 条件 | 处理 |
|---|---|---|
| Level 0 | 正常匹配 | 记录日志 |
| Level 1 | 疑似匹配 | 待复核 |
| Level 2 | 多帧未知 | 生成陌生人员事件 |
| Level 3 | 敏感区域多帧未知 | LED 提示，管理员复核 |
| Level 4 | 黑名单人员或高风险规则命中 | LED + 蜂鸣器 + 管理端告警 |

### 12.2 去重规则

- 同一人员同一摄像头在 10 分钟内只生成一条晚归事件。
- 同一未知目标同一摄像头在 5 分钟内只生成一条陌生人员事件。
- 同一 tracking_id 不重复生成同类事件。

## 13. 隐私与合规

必须满足以下要求：

- 采集前取得明确授权。
- 明确告知采集目的、使用范围、保存期限。
- 只保存必要字段。
- 管理员访问需要权限控制。
- 人脸原图和抓拍图限制访问。
- 毕业或离院后按制度删除或失效人员数据。
- 陌生人抓拍默认 7 天后删除。
- 所有管理员操作记录审计日志。
- 晚归记录需要人工复核后再作为正式依据。

## 14. 迭代计划

### 14.1 第一阶段：原型验证

目标：跑通识别主链路。

功能：

- 摄像头采集。
- InsightFace 人脸识别。
- SQLite 人员库。
- 人员注册。
- 实时比对。
- 识别日志。

验收标准：

- 能注册人员。
- 摄像头前能识别已注册人员。
- 未注册人员能生成未知记录。

### 14.2 第二阶段：试点可用

目标：降低误报，形成管理闭环。

功能：

- 多角度注册。
- 人脸质量评估。
- 多帧投票。
- 陌生人员事件。
- 晚归疑似事件。
- 管理员复核。
- 配置化阈值。

验收标准：

- 常见光照下识别稳定。
- 低质量人脸不会直接触发强告警。
- 管理员能复核事件。

### 14.3 第三阶段：正式部署

目标：接入实际业务数据。

功能：

- 接入门禁记录。
- 进出方向判断。
- 分级告警。
- 数据导出。
- 权限管理。
- 自动清理过期数据。

验收标准：

- 晚归事件能结合出入记录判断。
- 陌生人员事件误报率可控。
- 数据访问和删除符合管理要求。

## 16. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 光照差 | 识别率下降 | 补光、WDR 摄像头、CLAHE |
| 侧脸和遮挡 | 匹配失败 | 多角度注册、质量评估、等待正脸 |
| 误报陌生人 | 干扰管理 | 多帧投票、分级告警、人工复核 |
| 晚归误判 | 管理争议 | 接入门禁记录、人工复核 |
| 算力不足 | 延迟升高 | 降低检测频率、跟踪复用、限制处理人数 |
| 隐私风险 | 合规问题 | 授权、最小化、权限控制、定期删除 |

## 18. 验收标准

项目初期验收：

- 可成功注册人员信息和人脸特征。
- 可从摄像头实时读取画面。
- 可识别已注册人员。
- 可记录识别日志。
- 可生成陌生人员事件。
- 可根据时间生成晚归疑似事件。
- 阈值、门禁时间、去重窗口可配置。
- 低置信度结果进入复核而不是直接强告警。

试点验收：

- 连续运行 4 小时无崩溃。
- 摄像头断开后可恢复或提示异常。
- 管理员可查询和复核事件。
- 误报事件可被标记为驳回。
- 抓拍图和日志按保留策略清理。
