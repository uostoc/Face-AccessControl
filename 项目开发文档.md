# 人脸识别门禁与晚归管理系统开发文档

## 1. 项目概述

本项目用于书院场景下的人员识别、陌生人员发现和晚归事件辅助管理。系统通过摄像头采集视频流，对画面中的人脸进行检测、对齐、特征提取和人员库比对，最终生成通行日志、陌生人员事件和晚归疑似记录。

项目定位为“辅助识别与管理系统”，不建议在初期直接作为强制处罚或自动处置依据。所有低置信度、陌生人、晚归疑似事件应支持管理员复核。

## 2. 建设目标

### 2.1 核心目标

- 识别书院合法人员。
- 发现非书院人员或未知人员。
- 记录人员在指定区域的出现时间、地点和抓拍证据。
- 在门禁时间后生成晚归疑似记录。
- 提供管理员复核、查询、导出和人员库维护能力。

### 2.2 非目标

- 初期不做自动处分。
- 初期不强依赖远程推送。
- 初期不直接接管物理门禁开关。
- 初期不承诺 100% 准确识别所有侧脸、遮挡、口罩、弱光场景。

## 3. 业务范围

### 3.1 非书院人员识别

系统检测到人脸后，与人员库进行比对。

- 高置信匹配：记录正常通行。
- 低置信匹配：进入观察或待复核。
- 多帧确认未知：生成陌生人员事件。
- 高风险场景下可触发本地 LED 或蜂鸣器提示。

### 3.2 晚归人员识别

系统在门禁时间后识别到人员时，生成晚归相关记录。

推荐分阶段实现：

- V1：仅生成“晚归疑似记录”，由管理员复核。
- V2：接入门禁出入记录，判断是否属于“外出后超时返回”。
- V3：通过双摄像头或方向识别判断进出方向。

### 3.3 人员库管理

支持常驻学生、老师、管理员、访客、失效人员的统一维护。

人员状态：

- `active`：有效人员。
- `visitor`：临时访客。
- `expired`：已过期人员。
- `graduated`：已毕业人员。
- `disabled`：停用人员。

## 4. 总体架构

```text
摄像头 / 机器人
    ↓
视频采集服务
    ↓
人脸检测与质量评估
    ↓
人脸对齐与特征提取
    ↓
人员库比对
    ↓
事件判定引擎
    ↓
SQLite 数据库 / 文件存储
    ↓
管理端 / 日志 / 告警 / 报表
```

## 5. 初步设计要求

### 5.1 设计目标

初步设计阶段需要明确系统的整体能力边界，保证后续开发可以围绕稳定识别、可追溯记录和人工复核展开。

- 系统应支持本地摄像头或机器人摄像头视频输入。
- 系统应能对画面中的人脸进行检测、质量评估、特征提取和人员库比对。
- 系统应能记录正常通行、疑似匹配、陌生人员、晚归疑似等事件。
- 系统应提供人工复核能力，低置信度结果不得直接作为最终结论。
- 系统应满足人脸数据采集授权、访问控制、日志留痕和数据保留期限要求。
- 系统应具备后续扩展能力，可接入门禁记录、方向判断、远程告警和向量检索服务。

### 5.2 设计原则

- **模块化**：视频采集、人脸识别、人员比对、事件判定、数据存储和管理端应解耦设计。
- **配置化**：识别阈值、门禁时间、检测频率、事件去重窗口、告警开关应通过配置文件或配置表维护。
- **低误报优先**：低质量人脸、侧脸、遮挡、弱光场景不直接触发强告警，应进入等待、观察或复核流程。
- **人工复核优先**：陌生人员和晚归记录初期作为“疑似事件”，经管理员确认后再作为正式结果。
- **数据最小化**：只保存业务必要的身份字段、特征向量、识别日志和事件证据，避免存储无关个人信息。
- **可维护性**：数据库结构、目录结构、配置项和日志格式应保持清晰，便于后续调试和扩展。

### 5.3 功能设计要求

人员注册：

- 支持录入人员基础信息，包括学号或工号、姓名、角色、院系、状态和有效期。
- 支持采集或上传正脸、左侧、右侧注册照片。
- 注册照片必须经过质量评估，质量不合格时不生成正式特征。
- 人脸特征应与人员信息关联保存。

实时识别：

- 支持从摄像头持续读取视频帧。
- 支持对每帧或按固定频率抽帧进行人脸检测。
- 支持对合格人脸提取特征并与人员库比对。
- 支持生成识别日志，记录时间、地点、摄像头、相似度、匹配结果和抓拍图。

陌生人员识别：

- 未匹配人员不应依赖单帧结果直接告警。
- 系统应结合人脸质量、多帧投票和事件去重生成陌生人员事件。
- 陌生人员事件应支持管理员复核。
- 初期采用分级告警策略，不默认启用强蜂鸣报警。

晚归识别：

- 系统应支持配置门禁时间和宽限期。
- 初期只生成“晚归疑似”记录。
- 晚归疑似记录应包含人员、时间、地点、摄像头和抓拍证据。
- 管理员复核后确认或驳回晚归事件。

事件复核：

- 管理员应能查看陌生人员事件和晚归疑似事件。
- 管理员应能将事件标记为确认或驳回。
- 复核结果应保存复核人、复核时间和备注。

日志查询：

- 支持按时间、人员、摄像头、地点和事件类型查询识别日志。
- 支持查询正常通行、疑似匹配、陌生人员和晚归疑似记录。

系统配置：

- 支持配置识别确认阈值、疑似阈值、多帧投票窗口、门禁时间、宽限期、去重窗口和告警开关。
- 配置变更应记录更新时间，后续可扩展管理员操作审计。

### 5.4 性能设计要求

- 摄像头输入推荐使用 720P 或 1080P，帧率 30FPS。
- 人脸检测频率建议控制在 5 到 10FPS，避免逐帧检测造成算力浪费。
- 单人近距离正常光照场景下，应实现接近实时识别。
- 多人同时出现时，应优先处理画面中面积最大的前 3 张人脸。
- 同一跟踪目标不应每帧重复提取特征，建议按 1 到 2 秒间隔复用识别结果。
- 摄像头断开后应能提示异常，并按配置间隔尝试重连。
- 系统应支持连续运行测试，初期验收目标为连续运行 4 小时无崩溃。

### 5.5 数据设计要求

- 初期数据库采用 SQLite。
- 数据库至少包含人员表、特征表、识别日志表、安全事件表和系统配置表。
- 人员表保存人员基础信息、角色、状态、有效期和授权状态。
- 特征表保存注册照片路径、512 维人脸特征、拍摄角度、质量评分和模型名称。
- 识别日志表保存每次识别结果，便于追溯和调试。
- 安全事件表保存陌生人员、晚归疑似、摄像头异常等需要处理的事件。
- 系统配置表保存阈值、时间窗口、告警开关等可变参数。
- 抓拍图片和注册照片以文件形式保存，数据库仅保存路径和必要元数据。

### 5.6 安全与合规设计要求

- 人脸数据采集前应取得明确授权，并说明采集目的、使用范围和保留期限。
- 系统应限制人脸数据、抓拍图和识别日志的访问权限。
- 管理员复核、配置修改、人员数据变更等操作后续应支持审计留痕。
- 陌生人员抓拍图应设置保留期限，默认 7 天后清理。
- 毕业、离院或访客过期人员应按制度删除或失效其人脸数据。
- 晚归记录初期必须经过人工确认，不应直接作为自动处罚依据。
- 系统应遵循数据最小化原则，避免保存与识别和管理无关的个人信息。

### 5.7 异常处理要求

- 摄像头断开：记录摄像头异常事件，并按配置间隔重连。
- 低质量人脸：记录为质量不足或跳过识别，不直接判定为陌生人员。
- 模型加载失败：启动时给出明确错误信息，禁止进入半可用识别状态。
- 数据库异常：记录错误并停止写入相关业务事件，避免生成不完整数据。
- 重复事件：同一人员或同一未知目标在配置时间窗口内不重复生成同类事件。
- 配置缺失：使用文档约定的默认值启动，并在日志中提示缺失配置。

### 5.8 可扩展要求

- 后期可接入门禁刷卡或闸机记录，提高晚归判断准确性。
- 后期可通过双摄像头、区域线或运动方向判断进出方向。
- 人员规模扩大后，可引入 FAISS 向量库替代 SQLite 中的全量相似度遍历。
- 告警模块后期可扩展远程推送、短信、企业微信或管理端实时通知。
- 管理端后期可增加管理员权限分级、操作审计和数据导出审批。
- 算力不足时可替换轻量模型或引入边缘计算设备。

## 6. 技术选型

| 模块 | 推荐方案 | 说明 |
|---|---|---|
| 开发语言 | Python | 适合 OpenCV、InsightFace、模型推理 |
| 人脸识别框架 | InsightFace | 集成检测、关键点、特征提取 |
| 模型包 | `buffalo_l` | 初期综合效果较好 |
| 推理引擎 | onnxruntime | InsightFace 常用依赖 |
| 视频处理 | opencv-python | 摄像头采集、图像处理 |
| 数值计算 | numpy | 向量计算、余弦相似度 |
| 数据库 | SQLite | 初期比 JSON 更适合审计、查询和维护 |
| 后期向量检索 | FAISS | 人数规模增大后引入 |
| 管理端 | FastAPI + Web 前端 | 可分阶段建设 |

## 7. 模块划分

### 7.1 视频采集模块

职责：

- 打开 USB 摄像头、网络摄像头或机器人视频流。
- 按配置读取视频帧。
- 支持摄像头离线重连。
- 记录摄像头状态。

关键配置：

```yaml
camera:
  source: 0
  fps: 30
  width: 1280
  height: 720
  reconnect_interval_seconds: 5
```

### 7.2 人脸检测模块

职责：

- 检测画面中的所有人脸。
- 输出人脸框、关键点、检测置信度。
- 过滤过小、模糊、角度过大的无效人脸。

建议策略：

- 检测频率控制在 5 到 10 FPS。
- 中间帧使用跟踪结果复用。
- 同一帧只处理面积最大的前 N 张人脸。

### 7.3 人脸质量评估模块

职责：

- 判断人脸是否适合用于识别。

质量指标：

- 人脸面积。
- 清晰度。
- 亮度。
- 姿态角。
- 遮挡情况。
- 是否单人脸。

低质量人脸不直接判定为陌生人，只记录为“质量不足”或等待下一帧。

### 7.4 特征提取模块

职责：

- 对合格人脸进行对齐。
- 提取 512 维特征向量。
- 对向量进行归一化。

输出：

```text
embedding: float[512]
quality_score: float
face_image_path: string
```

### 7.5 人员库比对模块

职责：

- 从数据库加载有效人员特征。
- 使用余弦相似度计算最相似人员。
- 返回匹配结果、相似度和候选人员。

初始阈值：

| 相似度 | 判定 |
|---|---|
| `>= 0.60` | 确认匹配 |
| `0.40 ~ 0.60` | 疑似匹配 |
| `< 0.40` | 未匹配 |

阈值必须支持配置，并在现场试运行后校准。

### 7.6 多帧投票模块

职责：

- 降低单帧误识别。
- 对同一个跟踪目标累计多次识别结果。

推荐规则：

```text
连续 5 次有效识别中，至少 3 次匹配同一人员，确认身份。
连续 5 次有效识别中，至少 3 次为未知，生成陌生人员事件。
```

### 7.7 事件判定模块

职责：

- 将识别结果转换为业务事件。

事件类型：

- 正常通行。
- 疑似匹配。
- 陌生人员。
- 晚归疑似。
- 质量不足。
- 摄像头异常。

### 7.8 告警模块

职责：

- 对高风险事件进行本地提示或管理端提示。

初期建议：

- 正常通行：只记录日志。
- 疑似匹配：进入复核列表。
- 陌生人员：记录事件，可 LED 提示。
- 高风险陌生人员：LED + 蜂鸣器。

不建议初期对所有陌生人直接蜂鸣器报警。

### 7.9 管理端模块

最低功能：

- 人员列表。
- 人员新增、编辑、停用。
- 注册照上传或采集。
- 人脸特征生成。
- 通行日志查询。
- 陌生人员事件复核。
- 晚归疑似事件复核。
- 阈值配置。
- 摄像头状态查看。
- 数据导出。

## 9. 核心流程

### 9.1 人员注册流程

```text
录入人员基础信息
    ↓
采集正脸、左侧、右侧照片
    ↓
人脸质量检测
    ↓
质量合格则提取特征
    ↓
写入人员表和特征表
    ↓
注册完成
```

### 9.2 实时识别流程

```text
读取视频帧
    ↓
人脸检测
    ↓
质量评估
    ↓
合格人脸提取特征
    ↓
人员库相似度比对
    ↓
多帧投票
    ↓
生成识别日志
    ↓
按规则生成业务事件
```

### 9.3 陌生人员事件流程

```text
检测到未知人脸
    ↓
确认人脸质量合格
    ↓
多帧投票仍为未知
    ↓
检查是否短时间重复事件
    ↓
生成陌生人员事件
    ↓
本地提示或管理员复核
```

### 9.4 晚归事件流程

```text
识别到有效人员
    ↓
获取当前时间
    ↓
判断是否超过门禁时间和宽限期
    ↓
检查短时间内是否已记录
    ↓
生成晚归疑似事件
    ↓
管理员复核
```

## 13. 告警策略

### 13.1 分级规则

| 等级 | 条件 | 处理 |
|---|---|---|
| Level 0 | 正常匹配 | 记录日志 |
| Level 1 | 疑似匹配 | 待复核 |
| Level 2 | 多帧未知 | 生成陌生人员事件 |
| Level 3 | 敏感区域多帧未知 | LED 提示，管理员复核 |
| Level 4 | 黑名单人员或高风险规则命中 | LED + 蜂鸣器 + 管理端告警 |

### 13.2 去重规则

- 同一人员同一摄像头在 10 分钟内只生成一条晚归事件。
- 同一未知目标同一摄像头在 5 分钟内只生成一条陌生人员事件。
- 同一 tracking_id 不重复生成同类事件。

## 14. 隐私与合规

必须满足以下要求：

- 采集前取得明确授权。
- 明确告知采集目的、使用范围、保存期限。
- 只保存必要字段。
- 管理员访问需要权限控制。
- 人脸原图和抓拍图限制访问。
- 毕业或离院后按制度删除或失效人员数据。
- 陌生人抓拍默认 7 天后删除。
- 所有管理员操作记录审计日志。
- 晚归记录需要人工复核后再作为正式依据。

## 17. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 光照差 | 识别率下降 | 补光、WDR 摄像头、CLAHE |
| 侧脸和遮挡 | 匹配失败 | 多角度注册、质量评估、等待正脸 |
| 误报陌生人 | 干扰管理 | 多帧投票、分级告警、人工复核 |
| 晚归误判 | 管理争议 | 接入门禁记录、人工复核 |
| 算力不足 | 延迟升高 | 降低检测频率、跟踪复用、限制处理人数 |
| 隐私风险 | 合规问题 | 授权、最小化、权限控制、定期删除 |
